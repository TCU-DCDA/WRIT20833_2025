The Myth of Neutral Code

❌ "HTML/CSS are just technical tools"

This assumes code makes no claims about what matters, who's important, or how to understand information.

✓ Code Makes Arguments

Every technical choice you make is a rhetorical choice:

  • What you put in <h1> = "This is most important"
  • What you hide in <aside> = "This is secondary"
  • What you make 48px = "Pay attention here"
  • What you put first = "Start here"

Just like data has connotations, code has rhetoric.

Same Content, Different Arguments

Version 1: Academic Hierarchy

Large heading: "Research Methodology"
Small text: Community voices
Structure: Theory → Methods → Results → Communities

Argument: Expert knowledge is primary. Community input is supplementary.

Version 2: Community-Centered

Large heading: "Community Voices"
Small text: Research methodology
Structure: Voices → Context → Analysis → Technical Notes

Argument: Community knowledge is primary. Academic framing is secondary.

Version 3: Dialogic

Equal headings: "Research" + "Community"
Side-by-side layout
Structure: Conversation between voices

Argument: Multiple perspectives in dialogue. No single authority.

Same data. Same words. Different HTML/CSS structure = Different rhetorical claims about whose knowledge matters.

HTML as Classification Logic

Semantic Tags = Acts of Categorization

<header>

Says: "This content defines/introduces the site"

Power: You decide what defines your project

<main>

Says: "This is the central purpose"

Power: You decide what's primary vs. peripheral

<aside>

Says: "This is tangentially related"

Power: You decide what gets marginalized

<nav>

Says: "These are the key destinations"

Power: You decide what paths matter

Framework: Classification Logic

Remember: How you categorize shapes how people understand. HTML structure is an act of classification—and all classification involves power.

CSS as Visual Argument

Every Style Choice is a Claim

Font Size

font-size: 48px; = "This matters most"

font-size: 12px; = "This is less important"

Who decides what's important?

Color

color: red; = "Warning/urgency/danger"

color: green; = "Success/growth/affirmation"

Cultural associations shape meaning

Position

position: top; = "See this first"

display: none; = "This doesn't exist (for some audiences)"

Visibility = power

Space

margin: 100px; = "Give this breathing room"

margin: 0; = "Pack this tight"

Whitespace communicates value

Accessibility as Ethical Argument

Code Choices = Inclusion/Exclusion Decisions

Alt Text

<img src="chart.jpg" alt="">

Argument: "If you can't see, this insight isn't for you"

vs.

<img src="chart.jpg" alt="Bar chart showing 60% increase in mentions of climate change from 2010-2020">

Argument: "This insight belongs to everyone"

Color Contrast

color: #ddd; background: #fff; (low contrast)

Argument: "Only people with perfect vision matter"

Responsive Design

@media (max-width: 768px)

Argument: "Mobile users deserve the same experience"

Framework: Sacred Boundaries

Every access decision is a boundary decision. Who gets in? Who's left out? Accessible code is an ethical stance.

Code Rhetoric in Digital Humanities

Example 1: Digital Archive

Choice: Put artifact images in <main>, curator analysis in <aside>

Argument: Objects speak for themselves. Expert interpretation is supplementary.

Alternative: Interweave images and analysis in <article> sections

Argument: Objects and interpretation are inseparable.

Example 2: Topic Modeling Results

Choice: Large font for topic labels, small font for example documents

Argument: Patterns matter more than individual texts.

Alternative: Equal visual weight, side-by-side

Argument: Computational findings and textual evidence equally important.

Example 3: Data Story

Choice: Scroll-triggered CSS animations revealing data progressively

Argument: Understanding builds over time. Trust the narrative journey.

Alternative: All data visible immediately, navigable sections

Argument: Readers should explore at their own pace and priority.

From Coder to Rhetorician

❌ Old Mindset:

"I'm learning HTML/CSS to build websites"

"Code is technical and neutral"

"I just want it to work"

✓ New Mindset:

"I'm learning HTML/CSS to make public arguments"

"Every line of code is a rhetorical choice"

"I want it to work AND say what I mean"

Questions to Ask While Coding:

  • What does this tag/style choice claim is important?
  • Whose perspective is centered? Whose is marginalized?
  • Who can access this? Who's excluded?
  • What assumptions am I baking into the structure?
  • What alternative arguments could I make with different code choices?
